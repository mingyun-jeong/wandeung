# Flutter Design — 종합 디자인 스킬

## Overview

Flutter 앱의 디자인 작업을 체계적으로 수행하는 스킬. 세 가지 모드를 지원한다:

1. **Design-to-Code** — 디자인 아이디어, 스크린샷, 와이어프레임을 Flutter 위젯 코드로 변환
2. **UI/UX Review** — 기존 Flutter UI 코드를 검토하고 디자인 개선점을 제안
3. **Design System** — 컬러, 타이포그래피, 컴포넌트 토큰을 정의하고 관리

**Announce at start:** "flutter-design 스킬을 사용합니다. 모드: [Design-to-Code / UI/UX Review / Design System]"

## Mode Detection

사용자의 요청에서 모드를 자동 판별한다:

| 키워드/상황 | 모드 |
|------------|------|
| 스크린샷, 이미지, "이렇게 만들어줘", 와이어프레임, 디자인 시안 | Design-to-Code |
| "리뷰해줘", "개선점", "뭐가 문제야", 기존 화면 경로 지정 | UI/UX Review |
| "컬러 팔레트", "테마", "디자인 토큰", "컴포넌트 규칙" | Design System |
| 불분명 | 사용자에게 모드 선택 질문 |

## Mode 1: Design-to-Code

### Process

1. **입력 분석** — 스크린샷/와이어프레임/텍스트 설명을 분석하여 UI 요소 파악
2. **기존 코드 탐색** — 프로젝트에서 재사용 가능한 위젯, 테마, 상수 확인
3. **위젯 트리 설계** — 구조를 텍스트로 먼저 제시하고 사용자 확인
4. **코드 작성** — 프로젝트 컨벤션에 맞는 Flutter 코드 생성
5. **검증** — `flutter analyze` 통과 확인

### Design-to-Code 규칙

- **프로젝트 디자인 시스템 우선:** 하드코딩 색상/크기 대신 프로젝트의 테마 토큰과 상수를 사용
- **기존 위젯 재사용:** 새 위젯을 만들기 전에 `lib/widgets/`에서 유사한 위젯 검색
- **반응형 고려:** `MediaQuery`, `LayoutBuilder`, `Flexible/Expanded`를 활용하여 다양한 화면 크기 대응
- **성능 우선:**
  - 긴 리스트는 `ListView.builder` 또는 `SliverList`
  - 이미지는 `CachedNetworkImage` 또는 적절한 캐싱
  - 불필요한 rebuild 방지 (`const` 생성자, 위젯 분리)
- **접근성:** 터치 타겟 최소 48x48, `Semantics` 위젯 필요 시 추가
- **CLAUDE.md 디자인 규칙 준수:**
  - 카드/컨테이너 배경: `Color(0xFFF0F0F0)` 또는 `Colors.white` — Material colorScheme 배경색 사용 금지
  - 한국어 하드코딩 (i18n 없음)

### 위젯 트리 제시 형식

코드를 작성하기 전에 위젯 트리를 간결하게 제시한다:

```
Scaffold
├── AppBar(title: "화면 제목")
└── Column
    ├── _HeroCard (커스텀 위젯)
    │   ├── Text("제목")
    │   └── Row → [Icon, Text("수치")]
    ├── SizedBox(h: 16)
    └── Expanded
        └── ListView.builder → RecordCard (기존 위젯 재사용)
```

## Mode 2: UI/UX Review

### Process

1. **대상 파악** — 리뷰할 화면/위젯 파일 확인 (사용자가 지정하지 않으면 질문)
2. **코드 읽기** — 해당 파일 + 사용하는 위젯/테마 파일 전부 읽기
3. **체크리스트 점검** — 아래 체크리스트 항목별로 분석
4. **리포트 제출** — 문제점과 개선안을 우선순위별로 제시
5. **수정 여부 확인** — 사용자 승인 후 코드 수정

### UI/UX 리뷰 체크리스트

**레이아웃 & 구조:**
- [ ] 위젯 트리 깊이가 과도하지 않은가 (3단계 이상 중첩된 Container/Padding 등)
- [ ] 하드코딩된 크기 값이 여러 곳에 산재하지 않는가
- [ ] Overflow 위험이 있는 곳은 없는가 (긴 텍스트, 작은 화면)
- [ ] 스크롤 가능 영역이 적절한가

**시각 디자인:**
- [ ] 색상이 프로젝트 디자인 시스템과 일관되는가
- [ ] 여백(padding/margin)이 일관되는가 (8의 배수 권장)
- [ ] 텍스트 스타일이 테마 TextTheme을 사용하는가
- [ ] 카드/컨테이너 배경이 CLAUDE.md 규칙을 따르는가

**사용성:**
- [ ] 터치 타겟이 48x48dp 이상인가
- [ ] 로딩 상태가 표시되는가
- [ ] 빈 상태(empty state)가 적절한가
- [ ] 에러 상태가 사용자 친화적인가
- [ ] 중요 액션에 확인 다이얼로그가 있는가

**성능:**
- [ ] `const` 생성자가 가능한 곳에 사용되었는가
- [ ] 리스트가 `builder` 패턴을 사용하는가
- [ ] 불필요한 `setState` 또는 provider rebuild가 없는가
- [ ] 이미지/비디오에 적절한 캐싱/로딩 전략이 있는가

**코드 품질:**
- [ ] 위젯이 적절히 분리되어 있는가 (한 파일이 300줄 이상이면 분리 고려)
- [ ] 매직 넘버가 상수로 추출되어 있는가
- [ ] 재사용 가능한 패턴이 위젯으로 추출되어 있는가

### 리포트 형식

```markdown
## UI/UX 리뷰: [화면 이름]

### 🔴 즉시 수정 (사용성/접근성 문제)
1. **[문제]** — 설명 → **수정안:** 코드 또는 접근법

### 🟡 개선 권장 (디자인/일관성)
1. **[문제]** — 설명 → **수정안:** 코드 또는 접근법

### 🟢 참고 (성능/코드 품질)
1. **[문제]** — 설명 → **수정안:** 코드 또는 접근법

### ✅ 잘된 점
- ...
```

## Mode 3: Design System

### Process

1. **현재 상태 파악** — `app.dart`의 ThemeData, `constants.dart`, 기존 위젯들의 스타일 패턴 분석
2. **토큰 정의/수정** — 사용자 요청에 따라 디자인 토큰 추가/수정
3. **영향 범위 확인** — 변경이 기존 화면에 미치는 영향 분석
4. **적용** — 테마 파일 수정 + 하드코딩된 값들을 토큰으로 교체

### 디자인 시스템 관리 영역

**컬러 토큰:**
- Primary, Secondary, Accent 등 시맨틱 컬러
- 상태 컬러 (Success, Warning, Error, Info)
- 표면 컬러 (Background, Surface, Card)
- 난이도 컬러 (DifficultyColor enum과 연동)

**타이포그래피:**
- TextTheme 전체 스케일 정의
- 폰트 패밀리 (한국어 최적화 폰트)
- line height, letter spacing 규칙

**간격 & 크기:**
- 여백 시스템 (4/8/12/16/20/24/32/48)
- 모서리 둥글기 (borderRadius) 규칙
- 아이콘 크기 규칙
- 터치 타겟 최소 크기

**컴포넌트 토큰:**
- 카드 스타일 (배경, 그림자, 둥글기, 여백)
- 버튼 스타일 (크기 변형: small/medium/large)
- 입력 필드 스타일
- 칩/뱃지 스타일
- BottomSheet 스타일

### 디자인 시스템 문서화

변경 시 아래 형식으로 요약하여 사용자에게 제시:

```
토큰 변경 요약
━━━━━━━━━━━━━━━━━━━━━
변경: [토큰 이름]
이전: [이전 값]
이후: [새 값]
영향: [영향받는 파일 수]개 파일
  - file1.dart (Line X)
  - file2.dart (Line Y)
━━━━━━━━━━━━━━━━━━━━━
```

## Cross-Mode Rules

모든 모드에서 공통으로 적용하는 규칙:

1. **CLAUDE.md 디자인 규칙은 절대 준수** — `colorScheme.surfaceContainerHighest` 등 Material colorScheme 배경색 사용 금지. 카드/컨테이너 배경은 `Color(0xFFF0F0F0)` 또는 `Colors.white`
2. **한국어 UI** — 모든 사용자 대면 텍스트는 한국어로 하드코딩
3. **Riverpod 패턴** — `ConsumerWidget`/`ConsumerStatefulWidget` 사용
4. **기존 코드 스타일 존중** — 새로운 패턴을 도입하기 전에 기존 코드의 패턴을 따름
5. **최소 변경 원칙** — 요청된 것만 변경. 주변 코드 리팩터링 금지
6. **코드 작성 전 확인** — 위젯 트리(Mode 1), 리포트(Mode 2), 토큰 요약(Mode 3)을 먼저 보여주고 사용자 승인 후 코드 작성

## Transition

- 디자인 작업이 브레인스토밍이 필요한 규모라면 → `brainstorming` 스킬로 전환
- 구현 계획이 필요한 규모라면 → 디자인 완료 후 `writing-plans` 스킬로 전환
- 작은 수정(단일 위젯, 색상 변경 등)은 이 스킬 안에서 바로 코드 작성까지 완료
